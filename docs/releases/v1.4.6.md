# Release v1.4.6

[![GitHub Downloads](https://img.shields.io/github/downloads/alexdelprete/ha-abb-fimer-pvi-vsn-rest/v1.4.6/total?style=for-the-badge)](https://github.com/alexdelprete/ha-abb-fimer-pvi-vsn-rest/releases/tag/v1.4.6)

**Release Date:** TBD

**Type:** Minor feature release - Known devices persistence, partial discovery repair notifications,
automatic re-discovery, and device sync tightening.

## What's Changed

### Added

- **Known devices persistence** - The integration remembers all discovered devices across
  restarts in `config_entry.data["known_devices"]`. This enables detection of missing devices
  when the system starts with partial availability (e.g., datalogger online but inverter
  still powering up at sunrise).

- **Partial discovery repair notification** - When known devices are missing at startup, a
  WARNING-level repair notification lists the missing devices with possible causes. The
  notification clears automatically when all devices recover.

- **Automatic re-discovery** - The coordinator runs `discover_vsn_device()` every poll cycle
  (default: 60s) when devices are missing. When recovered devices are found, the integration
  auto-reloads to create their entities — no manual reload needed.

- **Idempotent device detection** - If normalized data contains device IDs not in the known
  devices list (e.g., a deleted device still physically connected, or a new device), the
  integration triggers a reload for automatic device creation.

- **Datalogger livedata_device_id** - Added `livedata_device_id` field to `DiscoveredDevice`
  for consistent device matching when the livedata key differs from device_id (e.g., MAC
  address vs serial number for VSN300 dataloggers).

### Changed

- **Device deletion always allowed** - `async_remove_config_entry_device()` now always allows
  deletion. Deleted devices are removed from the `known_devices` list. If the device still
  physically exists, the next discovery will find it and re-add it. To permanently hide a
  device, disable it instead of deleting it.

- **Removed automatic orphan cleanup** - `_cleanup_orphaned_devices()` has been removed.
  Devices are never auto-deleted. This prevents the scenario where a temporarily offline
  inverter gets deleted at startup, losing all its entities and history.

- **Client device sync on re-discovery** - After re-discovery, the coordinator calls
  `client.update_discovered_devices()` to keep the client's device list and device_type
  injection map in sync immediately (previously stale until config reload).

- **Device type map optimized** - Built lazily on first poll as a dict lookup instead of
  iterating discovered devices every cycle. Invalidated on re-discovery.

- **Known devices device_type sync** - Existing entries with stale device_type (e.g.,
  `"unknown"` from v8 migration) are updated from discovery results on each startup.

### Refactored

- **client.py** - Extracted `update_discovered_devices()`, `_ensure_device_type_map()`, and
  `_inject_device_types()` helpers for clear device type injection flow.

- **coordinator.py** - Extracted `_update_discovery_state()`, `_sync_known_devices()`,
  `_handle_full_recovery()`, `_handle_partial_recovery()` from `_attempt_rediscovery()`.

- **\_\_init\_\_.py** - Extracted `_sync_known_devices()` and `_detect_missing_devices()` from
  `async_setup_entry()`. Setup flow now reads: sync → detect → notify.

### Migration

- **Config entry v7 → v8** - Populates `known_devices` from the existing device registry.
  Device type is inferred from model field (datalogger detection) and updated from discovery
  on first successful startup.
