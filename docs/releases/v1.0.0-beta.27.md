## Bug Fixes

### üêõ system_load - Fixed Excessive Decimal Places

**Problem**: The `system_load` sensor was displaying values with excessive decimal precision:

```text
Current: 0.05419921875
Expected: 0.05
```text

**Root Cause**:

- Beta.21 attempted to fix this using `suggested_display_precision = 2`
- However, this property doesn't work reliably for sensors **without units**
- The `system_load` sensor has `unit_of_measurement: ""` (empty) in the mapping file
- Home Assistant ignores display precision hints when there's no unit

**Solution**: Instead of relying on `suggested_display_precision`, we now **round the value** in the `native_value` property before returning it.

**Code Change** ([sensor.py:446-448](d:%5COSILifeDrive%5CDev%5CHASS%5Cha-abb-fimer-pvi-vsn-rest%5Ccustom_components%5Cabb_fimer_pvi_vsn_rest%5Csensor.py#L446-L448)):

```python
# Round system_load to 2 decimals (suggested_display_precision not working without units)
if self._point_name == "system_load" and isinstance(value, (int, float)):
    return round(value, 2)
```text

**Result**:

- ‚úÖ Value stored in state: `0.05`
- ‚úÖ Displayed in UI: `0.05`
- ‚úÖ Proper precision maintained across all platforms

### üßπ model - Strip Leading/Trailing Dashes

**Problem**: The `model` (Model identifier) sensor was showing values with leading/trailing dashes:

```text
Current: -3G82-
Expected: 3G82
```text

**Root Cause**:

- The normalizer has a `STRING_STRIP_POINTS` registry for cleaning up string values
- This registry already includes `"pn"` (product number)
- However, it was missing `"C_Md"` (the VSN REST API name for model identifier)

**Solution**: Add `"C_Md"` to the `STRING_STRIP_POINTS` registry. The existing `strip("-")` logic handles the cleanup.

**Code Change**
([normalizer.py:38-41](d:%5COSILifeDrive%5CDev%5CHASS%5Cha-abb-fimer-pvi-vsn-rest%5Ccustom_components%5Cabb_fimer_pvi_vsn_rest%5Cabb_fimer_vsn_rest_client%5Cnormalizer.py#L38-L41)):

```python
# String cleanup registry: Points that need value cleanup
STRING_STRIP_POINTS = {
    "pn",    # Product number - strip leading/trailing dashes
    "C_Md",  # Model identifier - strip leading/trailing dashes
}
```text

**Result**:

- ‚úÖ Leading dashes removed: `-3G82-` ‚Üí `3G82`
- ‚úÖ Trailing dashes removed: `-3G82-` ‚Üí `3G82`
- ‚úÖ Internal dashes preserved: `PVI-10.0-OUTD` ‚Üí `PVI-10.0-OUTD` (unchanged)

**Note**: Python's `str.strip("-")` only removes leading and trailing characters, not internal ones. This is the desired behavior.

### üîß Datalogger device_type - Fixed "unknown"

**Problem**: Datalogger entity attributes showed `device_type: unknown` instead of `device_type: datalogger`.

**Root Cause**:

- The datalogger device **doesn't appear in the VSN REST API `/v1/livedata` response**
- It only appears in `/v1/feeds` (with `"universal_logger"` description)
- The discovery code only processed devices from livedata, so it never saw the datalogger

**Solution**: Create a synthetic datalogger device from `status.json` data during discovery.

**Code Change**
([discovery.py:291-322](d:%5COSILifeDrive%5CDev%5CHASS%5Cha-abb-fimer-pvi-vsn-rest%5Ccustom_components%5Cabb_fimer_pvi_vsn_rest%5Cabb_fimer_vsn_rest_client%5Cdiscovery.py#L291-L322)):

```python
# Create synthetic datalogger device from status.json
keys = status_data.get("keys", {})
logger_sn = keys.get("logger.sn", {}).get("value", "Unknown")

# For VSN700, logger S/N might be in loggerId field (MAC address)
if logger_sn == "Unknown" and vsn_model == "VSN700":
    logger_sn = keys.get("logger.loggerId", {}).get("value", "Unknown")

# Clean logger ID (remove colons if MAC)
logger_id = logger_sn.replace(":", "") if ":" in logger_sn else logger_sn

datalogger_device = DiscoveredDevice(
    device_id=logger_id,
    raw_device_id=logger_sn,
    device_type="datalogger",  # ‚Üê Explicit type
    device_model=vsn_model,
    manufacturer="ABB" if vsn_model == "VSN300" else "FIMER",
    firmware_version=keys.get("fw.release_number", {}).get("value"),
    hardware_version=None,
    is_datalogger=True,
)
```text

**Result**:

- ‚úÖ Datalogger device properly identified with `device_type: "datalogger"`
- ‚úÖ Works for both VSN300 and VSN700
- ‚úÖ Uses existing status.json data (no extra API calls)

### üè∑Ô∏è VSN REST Name Attributes - Changed to "N/A"

**Problem**: Entity attributes showed `Vsn700 rest name: unknown` for VSN300 devices (and vice versa).

**Root Cause**: When a VSN REST API name wasn't applicable (None), the code used empty string as default, which Home Assistant displayed as "unknown".

**Solution**: Use `"N/A"` for None values instead of empty string.

**Code Change** ([sensor.py:482-483](d:%5COSILifeDrive%5CDev%5CHASS%5Cha-abb-fimer-pvi-vsn-rest%5Ccustom_components%5Cabb_fimer_pvi_vsn_rest%5Csensor.py#L482-L483)):

```python
# FROM:
"vsn300_rest_name": point_data.get("vsn300_name", ""),
"vsn700_rest_name": point_data.get("vsn700_name", ""),

# TO:
"vsn300_rest_name": point_data.get("vsn300_name") or "N/A",
"vsn700_rest_name": point_data.get("vsn700_name") or "N/A",
```text

**Result**:

- ‚úÖ VSN300 devices show `Vsn700 rest name: N/A` (clear and explicit)
- ‚úÖ VSN700 devices show `Vsn300 rest name: N/A`
- ‚úÖ Cleaner attribute display throughout
