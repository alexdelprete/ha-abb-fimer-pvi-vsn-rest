# Release v1.5.5

[![GitHub Downloads](https://img.shields.io/github/downloads/alexdelprete/ha-abb-fimer-pvi-vsn-rest/v1.5.5/total?style=for-the-badge)](https://github.com/alexdelprete/ha-abb-fimer-pvi-vsn-rest/releases/tag/v1.5.5)

**Release Date:** 2026-06-08

**Type:** Patch release - Energy sensor stale-value guard fixes.

## What's Changed

### Fixed

- **Periodic energy sensors stuck on `unknown` since v1.5.3.** (Reported in #61) The
  cold-restart stale-value guard introduced in v1.5.3 ("Guard 2") was gated on
  `accumulation_mode`, so it also caught **periodic** (today/week/month) and **runtime**
  counters that legitimately reset. After the inverter slept overnight the entity went
  `unknown`, the restored baseline stayed pinned at yesterday's end-of-day total, and the
  next morning's reset to ~0 was discarded on every poll — locking sensors like
  "Energy AC – Produced (Today)" on `unknown` for the entire day. A cold Home Assistant
  restart while the live value was below the last persisted value hit the same trap.
- The guard is now scoped to **lifetime** sensors only, restoring the documented v1.5.1
  design where `sensor_scope` (not `accumulation_mode`) drives the stale-value guard.
  Periodic and runtime counters reset freely again; Home Assistant handles their
  `total_increasing` resets natively. Thanks to @nonth for the detailed root-cause analysis.
- **Lifetime battery cycle counters (`chc`, `dhc`, `cycle_num`) are now guarded too.** They
  hit an `int()` early-return in `native_value` before the stale-value guard ran, so a
  cold-restart warm-up `0` could slip through as a meter reset. The guard now runs before
  display formatting, giving these counters the same protection as energy totals.

### Documentation

- Added a Troubleshooting entry for the rare case where a **lifetime** total sticks on
  `unknown` — a corrupt high reading poisoning the guard reference, or a genuine counter
  reset. The guard intentionally does not auto-recover (that would re-admit the energy
  double-count it prevents); the documented workaround is to reload the integration.
